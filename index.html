What is Ops-dashboard?

Currently the Operations side of the IT industry is lacking a good dashboard implementation that allows displaying alert data in a meaningful way on a big display.

This project aims to fill this gap, using an HTML/Javascript/PHP based solution that gets it's information from Icinga/Nagios's JSON output possibility.

Goals of a/the dashboard

Such a dashboard can be useful in multiple use cases, by primary goal is to be used as a wallboard, displayed on a big LCD screen in the operations team's office, however there's a valid need for an in-browser use case as well, allowing a quick overview of the system, even from remote locations.

The Ops-dashboard is organized around tiles, with each tile representing a check/alert. The tiles are organized in a queue, which is wrapped to multiple columns and rows, depending on the size of the browser window and the number of alerts that currently needs to be displayed.

The dashboard is using a completely dynamic layout using DOM manipulation, therefore it requires no refreshes of the browser page to display information. As a service returns to an OK state, the tile is removed using an animation sequence which allows the Operations staff to better understand how the states are changing in their systems.

Features

Wallboard display

  • Information is highly visible by automatically scaling to target resolution
  • dynamic layout and animations show to visualize the state changes of the services, no page refresh needed
  • Show On-call/admin of the day in the header
  • Track when the last "all good" state occured
  • Team members may send messages to be displayed at the bottom of the page
  • For each alert, show:
    • Hostname on which the alert occured
    • Service name
    • Priority, if present
    • Service/Host checks, critical/warning/unknown states based on coloring of the tiles
    • In the form of tags:
    • Soft/Hard state
    • how long the alert is present
    • is the service flapping?

In-browser display

  • Fix the number of columns to a pre-defined value to get an overview of what is going on in the system
  • Scales automatically to mobile devices - needs some work though

Why not show the Icinga/Nagios dashboard page?

Many teams decide to put the icinga dashboard page on their big display, however due to the limitations of that interface it is not really good for showing much information. It refreshes every 60-90 seconds, there is too much information displayed, and when it comes to dashboards, less is definitely more. The font size is too small to see, and if you scale the page up , only a few services can be seen.

What technologies are used?

There is a set of PHP based backend scripts that fetch the JSON output of icinga/nagios. There is a HTML/JS (jQuery/jQuery-UI/Backbone/Underscore.js) based page that fetches the information from the PHP backend, and displays it.

Due to the same origin policy, the dashboard must be running from the same location as the icinga server's web interface is located on.
